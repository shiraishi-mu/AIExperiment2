5  pytest によるテスト駆動

システム開発では、部品として作る関数が意図した通りに動いていることが保障されていないと、組み上げたときに予期しない動作をする。 バグが発生してから手で動作確認をするのは手間がかかり、何度も繰り返し確認することはさらに大変である。 そこで、関数ごとに期待する入力と出力の組を書き、それを自動で検証するテストを用意する。 これを単体テスト（unit test） と呼ぶ。

ソフトウェア工学では、テストはおおまかに次の四段階に分類される。 本実験では、コード作成と直結する単体テストに集中する。

段階 概要
単体テスト 関数やクラスなど小さな単位の動作を確認
結合テスト 複数のモジュールをつなげて動作を確認
システムテスト 完成したシステム全体としての動作を確認
運用テスト 実際の利用環境で問題なく使えるかを確認

5.1 pytest のインストールと実行

本実験の Dev Container には、Python の標準的なテストフレームワークである pytest が予めインストールされている。 インストールされていない場合は、次のコマンドで導入できる。

pip install pytest

5.2 はじめてのテスト：assert 文

Python には assert 条件 という構文があり、条件が偽のときに AssertionError を発生させる。 これだけでも簡単なテストになる。

def square(x):
    return x * x

assert square(2) == 4
assert square(-3) == 9

スクリプトとして実行し、何も表示されなければテストは通っている。

5.3 pytest 形式のテスト

pytest では次のルールでテストを書く。

  • ファイル名は test_*.py または *_test.py
  • テスト関数の名前は test_ で始める
  • 関数内で assert を使う

例として、test_square.py を作って次のように書く。

# test_square.py
from mymath import square


def test_square_positive():
    assert square(2) == 4


def test_square_negative():
    assert square(-3) == 9


def test_square_zero():
    assert square(0) == 0

ターミナルで次のように実行する。

python -m pytest

すべてのテストが通れば次のような表示になる。

=========== 3 passed in 0.01s ===========

-v オプションをつけると、テスト関数ごとに結果が表示される。

python -m pytest -v

5.4 失敗するテストの読み方

square 関数を return x + x と間違えて書いてしまった場合、テストは失敗する。

FAILED test_square.py::test_square_positive - assert 4 == 4

実際の出力には、どの行で失敗したか、左辺と右辺がどのような値だったかが表示される。 この出力を見ながら、コードのどこが間違っているかを推測してデバッグする。

5.5 オセロのテスト例（先取り）

第6章で実装する initial_board() などの関数に対しては、次のようにテストを書く。 本書の各練習問題には動作確認用の assert 文がついているので、そのまま test_* 関数の中身に貼り付けてテストにできる。

# test_reversi.py
from reversi_yourid import (
    BOARD_SENTE, BOARD_GOTE, BOARD_EMPTY, BOARD_SIZE,
    initial_board, change_turn,
    board_movable, board_movable_any, board_move,
    board_number_check,
)


def test_initial_board():
    board = initial_board()
    assert board[3][3] == BOARD_GOTE
    assert board[3][4] == BOARD_SENTE
    assert board[4][3] == BOARD_SENTE
    assert board[4][4] == BOARD_GOTE
    assert board[0][0] == BOARD_EMPTY


def test_change_turn():
    assert change_turn(BOARD_SENTE) == BOARD_GOTE
    assert change_turn(BOARD_GOTE) == BOARD_SENTE


def test_initial_legal_moves():
    board = initial_board()
    moves = [
        (row, col)
        for row in range(BOARD_SIZE)
        for col in range(BOARD_SIZE)
        if board_movable(board, row, col, BOARD_SENTE)
    ]
    assert len(moves) == 4


def test_move_flips_stone():
    board = initial_board()
    board_move(board, 2, 3, BOARD_SENTE)
    assert board[2][3] == BOARD_SENTE
    assert board[3][3] == BOARD_SENTE

5.6 開発の流れ

本実験では、次の流れで開発を進めることを推奨する。

  1. 練習問題の関数のシグネチャだけ決めて、本体は pass で書く。
  2. その関数の動作確認用 assert を test_reversi.py に追加し、pytest で失敗することを確認する（Red）。
  3. 関数本体を実装し、pytest で通ることを確認する（Green）。
  4. コードを整理する（変数名、ループの書き方など）。テストが引き続き通ることを確認する（Refactor）。
  5. git add して git commit する。

これを Red-Green-Refactor サイクル と呼ぶ。 最初は冗長に感じるかもしれないが、慣れると「テストが通った」という確証を持って前に進めるようになる。

5.7 まとめ

  • 単体テストはコードの安心材料であり、後から壊してしまったときに気付くための仕組みでもある。
  • pytest は test_ で始まる関数の中で assert するだけで使える。
  • 本書の練習問題には動作確認用の assert がついているので、そのままテストに転用できる。
  • 各練習が終わるごとに、pytest でテストが通ることを確認してから git commit する。
