4  Gitによるバージョン管理

オセロゲームの実装を円滑に進めるため、二つのソフトウェア工学的な技術を習得する。 ひとつは本章で扱うGitによるバージョン管理、もうひとつは次章で扱うpytestによる単体テストである。 これらは、現代のソフトウェア開発では当たり前に使われている技術であり、人工知能のプログラミングを行う上でも避けては通れない。

なお本章は、慶應義塾大学の渡辺宙志先生のGitHub演習(渡辺宙志 2020)と、とほほのGit記事を参考にしている。 ここでは本実験で必要な最低限の機能だけを扱うので、より進んだ機能を学びたい人はこれらの資料を参照してほしい。

4.1 バージョン管理とは

プログラムに限らず、実験や研究は、これまでの学習や進捗を踏まえて次のステップに進んでいく営みである。 そのためにノートに記録をとり、何を行ったのか、何ができて何ができなかったのかを書き留めることが、効率的にプログラムや実験、研究を進めていく上で重要になる。

プログラミングは、目的の機能を実装するためにコードを書く作業である。 すべての機能を一度に実装することはできないので、骨格となるコードを書き上げ、さまざまな機能を一つずつ組み上げていく。 しかし、追加した機能が他のコードと相性が悪く全体としてバグが入る、ということは珍しくない。

このような問題に備え、うまく動いている状態でコードを保存しておき、新しく加えた機能で問題が起きたら元に戻れるようにする のが、バージョン管理の基本的な考え方である。 途中経過を逐次記録することで、問題への対処能力が大きく向上する。

最も素朴なバージョン管理は、ファイルをコピーして名前を変える方法である。 しかし、ファイル数が増えると管理が困難になり、どのコピーがどの状態なのか分からなくなる。 そこで本実験では、現在広く普及しているバージョン管理システムGitを用いる。

Note補足：分散型と集中型

Git は分散型バージョン管理システムであり、サーバが無くてもローカルで履歴を管理できる。 これに対し、CVS や Subversion は集中型で、サーバ上の履歴に対して全員が読み書きする。 分散型は障害に強く、ブランチ操作が軽量という利点がある。

4.2 Git の準備

4.2.1 Git のインストール

Dev Container 内ではすでに Git がインストールされている。 バージョンを確認しよう。

git --version

4.2.2 Git の初期設定

コミットの作者として記録される名前とメールアドレスを設定する。 個人のGitHubアカウントがある場合は、それと一致させると後で便利である。

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

設定内容は次のコマンドで確認できる。

git config --global --list

4.3 Git の基本概念

Git では次の三つの場所を意識する。

  1. ワーキングツリー（working tree）：ファイルの実体。普段編集している場所。
  2. インデックス（index, staging area）：次のコミットに含めるファイルの仮置き場。
  3. リポジトリ（repository）：履歴が保存される場所。.git/ ディレクトリの中。

ファイルを編集してから履歴に残すまでの流れは次の通りである。

編集 → git add（インデックスに登録） → git commit（履歴に保存）

最初は混乱するが、実際に使いながら身につけていけばよい。 自転車と同じで、理論を読むよりも繰り返し使ってみることが上達の近道である。

4.4 Git の基本コマンド

以下では、reversi_yourid.py を編集していくシナリオを使って Git の基本を確認する。

4.4.1 git init：リポジトリを作る

新しくバージョン管理を始めるディレクトリで実行する。

cd lab_env
git init

.git という隠しディレクトリが作られ、ここに履歴が保存される。 配布される lab_env.zip にはすでに .git が含まれているので、改めて init する必要は無い。

4.4.2 git status：状態を確認する

git status

ワーキングツリーとインデックスの状態が表示される。 迷ったら、まずこれを実行する習慣をつけるとよい。

4.4.3 git add：インデックスに登録する

ファイルを次のコミットに含めるために登録する。

git add reversi_yourid.py

複数ファイルをまとめて追加する場合は次のようにする。

git add reversi_yourid.py test_reversi.py

4.4.4 git commit：履歴に保存する

インデックスの内容を、メッセージ付きで履歴に保存する。

git commit -m "implement initial board"

メッセージは「何をしたかが分かる短い英文または日本語」で書く。 よくある書き方は次のように動詞から始める形である。

  • implement initial board
  • add board_movable function
  • fix off-by-one in board_scan

4.4.5 git log：履歴を見る

git log
git log --oneline           # 1行ずつ表示
git log --oneline --graph   # 分岐をグラフ表示

4.4.6 git diff：変更を確認する

何を変えたのか、コミット前に確認する。

git diff               # ワーキングツリーとインデックスの差分
git diff --cached      # インデックスと最新コミットの差分
git diff HEAD          # ワーキングツリーと最新コミットの差分

4.4.7 git checkout / git restore：変更を取り消す

直前のコミット時点に戻したい場合は、

git restore reversi_yourid.py

を使う。古い git checkout -- reversi_yourid.py でも同じことができる。

Warning

git restore は 編集中の変更を捨てる コマンドである。 コミットしていない変更は失われるので注意する。

4.5 練習：実際に commit してみる

ここまで読んだら、実際に手を動かして確認しよう。

4.5.1 練習G1：初回コミットを作る

lab_env ディレクトリで以下を実行し、初期状態をコミットせよ。

git status
git add reversi_yourid.py
git status
git commit -m "initial commit"
git log --oneline

4.5.2 練習G2：変更を加えて差分を確認する

reversi_yourid.py の先頭にコメント行を1つ追加し、次を順に実行せよ。

git diff
git add reversi_yourid.py
git diff               # 何も表示されない
git diff --cached      # 差分が表示される
git commit -m "add header comment"

4.5.3 練習G3：間違いを取り消す

reversi_yourid.py を間違って消してしまった、という想定で次を試せ。

echo "broken" > reversi_yourid.py   # 中身を破壊
git status                          # 変更が検出される
git restore reversi_yourid.py       # 直前のコミットに戻す

4.6 推奨されるコミットの粒度

本実験では、各章の練習が終わるたびに git commit することを推奨する。 これにより、後でうまくいかなくなったときに、最後にうまく動いた状態に戻すことができる。

例えば、本書の第6章の練習13までを実装し終えたら、

git add reversi_yourid.py
git commit -m "implement human reversi game (ex.13)"

のようにコミットする。 最終レポートでは、git log の内容そのものが開発履歴として評価対象になるので、こまめにコミットを残すこと。

4.7 まとめ

  • Git は分散型バージョン管理システムであり、本実験では init / add / commit / status / log / diff / restore を中心に使う。
  • 各章の練習が終わるごとに commit する習慣をつける。
  • 後の章ではテストと組み合わせて、テストが通るようになったらコミットする、という流れで進めていく。
