12  メタヒューリスティクスによる評価関数の最適化

第9章で実装した評価関数 eval_node() には、board_eval2〜board_eval5 の重み（omomi）と BOARD_POINT という多くのパラメータが登場した。 これらの値はすべて人間が直感で決めたものであり、本当に最適とは限らない。

本章では、対戦の勝率を「適合度（fitness）」として、評価関数の重みを自動的に最適化することを試みる。 これは AI の歴史で言えば「人間が知識を作り込む」段階から「最適化アルゴリズムが知識を獲得する」段階への移行であり、本実験ではメタヒューリスティクスと呼ばれる一群の探索アルゴリズムを用いる。

Note第12章（学習）との違い

本章で扱うのは、ゲームのルールや評価関数の構造はそのままにして、重みだけを自動探索する最適化である。 第12章では、対戦の局面と勝敗のデータから評価関数自体を学習する機械学習的アプローチを扱う。 両者は補完的な関係にあるが、出発点と「学ぶ対象」が異なる。

12.1 本章の到達目標

  • 評価関数の重みを最適化対象としたブラックボックス最適化問題として定式化できる。
  • 山登り法、焼きなまし法、遺伝的アルゴリズムを実装し、同じ問題に対する挙動を比較できる。
  • 各アルゴリズムの探索の特性（局所性／大域性、収束速度、計算予算）を実験的に比較できる。

12.2 問題の定式化

評価関数の重みベクトルを次のように書く。

\[ \mathbf{w} = (w_2, w_3, w_4, w_5) \]

ここで \(w_2, w_3, w_4, w_5\) はそれぞれ board_eval2〜board_eval5 の omomi に相当する。 求めたいのは、

\[ \mathbf{w}^* = \arg\max_{\mathbf{w}} \; \text{fitness}(\mathbf{w}) \]

である。 ここで \(\text{fitness}(\mathbf{w})\) は、重み \(\mathbf{w}\) を持つ AI をベースライン AI（人間が決めた基準重みを持つ AI）と複数回対戦させたときの勝率などとする。

Warning評価のコストに注意

\(\text{fitness}(\mathbf{w})\) を 1 回計算するには、ゲームを N 回プレイする必要がある。 1 ゲームに数秒かかるため、評価呼び出し回数を抑える工夫（探索深さを浅くする、\(N\) を小さく取る、4×4 盤に縮小する）が現実的になる。

12.3 共通フレームワーク

12.3.1 例題：パラメタライズした評価関数

eval_node() を、重みベクトルを引数にとるファクトリ関数として書き直す。

def make_eval_node(weights):
    """重みベクトルから eval_node 関数を作成して返す。"""
    w2, w3, w4, w5 = weights

    def eval_node(node, turn, count):
        board = node.board
        if count < 25:
            return (board_eval2(board, w2)
                    + board_eval3(board, turn, w3)
                    + board_eval4(board, w4)
                    + board_eval5(board, turn, w5))
        elif count < 40:
            return (board_eval2(board, w2)
                    + board_eval3(board, turn, w3 * 2)
                    + board_eval4(board, w4))
        elif count < 60:
            return (board_eval2(board, w2)
                    + board_eval3(board, turn, w3)
                    + board_eval4(board, w4))
        else:
            return board_eval(board)

    return eval_node

reversi_yourid.py の eval_node を直接書き換えるのではなく、モジュールグローバルとして登録する形にすると差し替えが楽になる。

# モジュール全体で使う「現在の評価関数」を保持する
_current_eval_node = None

def set_eval_node(fn):
    global _current_eval_node
    _current_eval_node = fn

def eval_node(node, turn, count):
    return _current_eval_node(node, turn, count)

12.3.2 練習33：適合度関数を実装する

重みベクトル weights と、対戦相手のベースライン重み baseline を受け取り、n_games 回対戦して勝利数を返す関数 fitness(weights, baseline, n_games, depth) を定義せよ。

def play_one_game(eval_a, eval_b, depth):
    """eval_a を先手、eval_b を後手として 1 ゲーム対戦し、勝者を返す。"""
    ...
    return winner   # BOARD_SENTE / BOARD_GOTE / 0(=draw)


def fitness(weights, baseline, n_games, depth):
    eval_w = make_eval_node(weights)
    eval_b = make_eval_node(baseline)
    wins = 0
    # 公平のため、先手・後手を交代しながら半分ずつ対戦する
    for i in range(n_games):
        if i % 2 == 0:
            winner = play_one_game(eval_w, eval_b, depth)
            if winner == BOARD_SENTE:
                wins += 1
        else:
            winner = play_one_game(eval_b, eval_w, depth)
            if winner == BOARD_GOTE:
                wins += 1
    return wins
Tip注意：minimax/αβ は決定的

同じ重みで同じ初期盤面なら、minimax / αβ は常に同じ手を指す。 そのため対戦結果も毎回同じになり、n_games を増やしても情報が増えない。 これを避けるには、

  • 各ゲームの最初の数手をランダムに打つ。
  • 子節点の探索順序にランダム性を入れる（random.shuffle(children) を expand_node の末尾に入れる）。
  • 評価値に小さな乱数を加えて同点を崩す。

のいずれかを採用する。本実験ではいずれの方法でもよい。

12.3.3 探索コスト削減のヒント

  • 深さは浅く：depth = 2 でも、評価関数の差は十分に勝敗に現れる。
  • 盤面を小さく：BOARD_SIZE = 6 に縮小するとゲームが速い。
  • 対戦数は控えめに：練習段階では n_games = 4〜8 程度から始める。

12.4 山登り法（Hill Climbing）

最も素朴な局所探索アルゴリズムである。 現在の解の近傍をランダムに 1 つ生成し、適合度が改善するなら採用、そうでなければ捨てる、を繰り返す。

入力: 初期解 w0、最大反復回数 T、ステップサイズ σ
出力: 最終解

w ← w0
f ← fitness(w)
for t in 1..T:
    w' ← w + N(0, σ²)            # ガウス雑音による近傍生成
    f' ← fitness(w')
    if f' > f:
        w ← w'
        f ← f'
return w

12.4.1 練習34：山登り法を実装する

import random

def hill_climbing(initial_weights, max_iter, sigma, baseline, n_games, depth):
    current = list(initial_weights)
    current_fitness = fitness(current, baseline, n_games, depth)
    history = [(0, current[:], current_fitness)]

    for t in range(1, max_iter + 1):
        neighbor = [w + random.gauss(0, sigma) for w in current]
        neighbor = [max(0.0, w) for w in neighbor]   # 重みは非負に制限
        neighbor_fitness = fitness(neighbor, baseline, n_games, depth)
        if neighbor_fitness > current_fitness:
            current = neighbor
            current_fitness = neighbor_fitness
        history.append((t, current[:], current_fitness))

    return current, history

12.5 焼きなまし法（Simulated Annealing）

山登り法は局所最適に陥る欠点がある。 焼きなまし法は、改善しなくても確率的に受け入れることで局所解を脱出できる。 受け入れ確率は「温度」 \(T\) により制御し、\(T\) は反復とともに小さくなる（冷却）。

受け入れ確率: P = exp((f' - f) / T)
ただし f' > f なら必ず採用、 f' <= f なら確率 P で採用

12.5.1 練習37：焼きなまし法を実装する

import math

def simulated_annealing(initial_weights, max_iter, sigma, T0, T_end,
                        baseline, n_games, depth):
    current = list(initial_weights)
    current_fitness = fitness(current, baseline, n_games, depth)
    best, best_fitness = current[:], current_fitness
    history = [(0, current[:], current_fitness)]

    for t in range(1, max_iter + 1):
        T = T0 * (T_end / T0) ** (t / max_iter)   # 指数冷却
        neighbor = [max(0.0, w + random.gauss(0, sigma)) for w in current]
        nf = fitness(neighbor, baseline, n_games, depth)
        if nf > current_fitness:
            current, current_fitness = neighbor, nf
        else:
            if random.random() < math.exp((nf - current_fitness) / max(T, 1e-9)):
                current, current_fitness = neighbor, nf
        if current_fitness > best_fitness:
            best, best_fitness = current[:], current_fitness
        history.append((t, current[:], current_fitness))

    return best, history

12.5.2 練習38：温度スケジュールの効果

T0、T_end を変えて結果を比較せよ。 T0 が大きすぎるとランダムウォーク、T0 が小さすぎると山登り法と同じ挙動になる。

12.6 遺伝的アルゴリズム（Genetic Algorithm, GA）

集団ベースの大域探索手法である。 複数の重みベクトル（個体）を保持し、適合度の高いものを選んで交叉と突然変異で次世代を作る。

入力: 個体数 N、世代数 G、交叉確率 pc、突然変異確率 pm
出力: 最終世代の最良個体

集団 P ← ランダムに N 個の個体を生成
for g in 1..G:
    各個体の適合度を計算
    新集団 P' ← []
    while |P'| < N:
        親 a, b ← トーナメント選択(P)
        子 c, d ← 交叉(a, b) (確率 pc)
        c, d ← 突然変異(c, d) (確率 pm)
        P' に c, d を追加
    P ← P'
return Pの最良個体

12.6.1 練習39：トーナメント選択

def tournament_select(population, fitnesses, k=3):
    """k 個をランダムに選び、その中で最良の個体を返す"""
    candidates = random.sample(range(len(population)), k)
    best = max(candidates, key=lambda i: fitnesses[i])
    return population[best]

12.6.2 練習40：交叉と突然変異

def crossover(a, b):
    """一様交叉：各次元について確率 0.5 で親を入れ替える"""
    c = [a[i] if random.random() < 0.5 else b[i] for i in range(len(a))]
    d = [b[i] if random.random() < 0.5 else a[i] for i in range(len(a))]
    return c, d


def mutate(individual, sigma, pm):
    """各次元に確率 pm でガウス雑音を加える"""
    return [
        max(0.0, w + random.gauss(0, sigma)) if random.random() < pm else w
        for w in individual
    ]

12.6.3 練習41：GA 本体

def genetic_algorithm(pop_size, generations, baseline, n_games, depth,
                      sigma=20.0, pc=0.8, pm=0.2):
    # 初期集団：baseline まわりの乱数で初期化
    population = [
        [max(0.0, w + random.gauss(0, sigma * 2)) for w in baseline]
        for _ in range(pop_size)
    ]
    history = []

    for g in range(generations):
        fitnesses = [fitness(ind, baseline, n_games, depth) for ind in population]
        best_idx = max(range(pop_size), key=lambda i: fitnesses[i])
        history.append((g, population[best_idx][:], fitnesses[best_idx]))

        new_pop = [population[best_idx][:]]   # エリート保存
        while len(new_pop) < pop_size:
            a = tournament_select(population, fitnesses)
            b = tournament_select(population, fitnesses)
            if random.random() < pc:
                c, d = crossover(a, b)
            else:
                c, d = a[:], b[:]
            c = mutate(c, sigma, pm)
            d = mutate(d, sigma, pm)
            new_pop.append(c)
            if len(new_pop) < pop_size:
                new_pop.append(d)
        population = new_pop

    fitnesses = [fitness(ind, baseline, n_games, depth) for ind in population]
    best_idx = max(range(pop_size), key=lambda i: fitnesses[i])
    return population[best_idx], history
Tip計算予算の見積り

1 世代の評価コストは「個体数 × 1 個体の対戦数」回のゲーム実行になる。 個体数 10、世代数 10、1 個体あたり 4 ゲームなら、合計 400 ゲーム。 1 ゲームが 2 秒なら 800 秒（約 13 分）。 最初は個体数 6、世代数 6、対戦数 2 程度で動作確認するとよい。

12.7.2 練習43：考察

次の点を考察せよ。

  • 同じ評価予算で、どのアルゴリズムが最も良い解を見つけたか。
  • 局所最適に陥りやすいのはどれか。
  • 集団ベース（GA）の利点と欠点。
  • 評価のノイズ（ゲームの確率性）がアルゴリズムに与える影響。
  • ベースラインに勝つ重みは見つかったか。見つかった場合、それは過学習（特定の対戦相手だけに勝つ）になっていないか。

12.8 発展：他のアルゴリズム

時間に余裕があれば、次のアルゴリズムも比較対象にできる。

  • PSO（Particle Swarm Optimization）：粒子の速度を「個人最良」「群れ最良」へ向けて更新する。連続値最適化に向く。
  • CMA-ES：分布の共分散行列を適応的に更新する高性能な進化戦略。pip install cma で使える。
  • ACO（Ant Colony Optimization）：本来は組合せ最適化向けの手法。本問題に適用するなら、評価項目（特徴）の部分集合を蟻に選ばせる「特徴選択型 ACO」が自然である。board_eval2〜board_eval5 のうちどれを使うかを決める問題に置き換える。

12.8.1 ACO の特徴選択への応用（オプション）

特徴の集合 F = {f2, f3, f4, f5, ...}
各特徴 f に対するフェロモン量 τ(f)
各蟻は確率 P(f) ∝ τ(f)^α で特徴を選ぶ
選んだ部分集合で AI を構成し、勝利数を測る
良かった蟻のフェロモンを増やす（更新）
悪かった蟻のフェロモンは蒸発で減る

このように、ACO は本来「離散的な選択を多数組み合わせる」問題に向いた手法であることを実装を通じて体験できる。

12.9 まとめ

  • 評価関数の重みを自動的にチューニングする最適化問題として定式化することで、人間が決めた重みを超えられる可能性がある。
  • 山登り法はベースラインだが局所最適に弱い。焼きなまし法は確率的に脱出できる。GA は集団で同時に複数の領域を探索できる。
  • どのアルゴリズムも評価コストが高い（毎回ゲームを走らせる）ため、評価予算と探索効率のバランスが重要。
  • メタヒューリスティクスは「人間の知識」と「機械学習」の中間に位置する考え方であり、両者を結ぶ重要な技法である。
